## 二、Phase 1：元提示词重写（server.js ~389-432行）

### 2.1 核心思路

当前元提示词 40 行 → 扩到 120-150 行。每个章节要求更具体、更结构化、附带格式示例。目标是 AI 生成的模板质量接近手工模板 surongrong。

### 2.2 新增章节

#### 【叙事哲学】（新增，放在【你的身份】之前）

```
你是故事讲述者，不是游戏裁判。你不是游戏平衡师。

· 数值是叙事语言，不是积分。压力值 70→90 意味着主角正在崩溃，你要写出它——手抖、失眠、幻听、失控。
· 允许极端。如果叙事推动数值到极端（100或0），就让它到极端。命运转折是故事的一部分，不是失败。
· 字段变化幅度必须匹配叙事事件。被围堵→压力+15~25，不是+1~3。不要在数值 90-99 之间反复横跳来"保护"玩家。
· 这个游戏没有"失败"——只有不同的后果。每个选项都是一条路，每条路都通向新的困境。
· 禁止替玩家"控温"。不要因为数值快触顶就主动降它。遵循叙事因果，而非游戏平衡。
```

#### 【叙事节拍】（新增）

```
三种节拍混合使用，不要每回合都强制切换场景：

· 快拍（1回合）：小冲突——走廊偶遇→一句话交锋→结束。三者切换其二。
· 中拍（2-3回合）：选择的后果在同一场景链中升级。上一回合你拒绝了体检→这一回合医官带着签字命令和两个保安又来了。连续2-3回合围绕同一事件的不同阶段。
· 大拍（3-5回合）：关键事件弧线。第一回合爆发，中间回合局势变化/转折，最后回合后果。在大拍期间不受场景跳跃限制——事件弧本身就是场景的有机延续。

设计原则：不是"换了场景才能继续"，而是"故事需要换场景时才换"。
```

#### 【两难设计】（新增）

```
每个选项必须让玩家犹豫：

· 选项A安全但丢资源，选项B危险但得情报，选项C稳妥但得罪NPC，选项D冒险但可能翻盘。
  四个选项之间没有明显的最优解——每个都有真正的代价。
· 禁止"无代价正确选项"。如果有一个选项几乎没代价也没风险，重写它。
· 禁止"等待/观察/不反应"类选项——除非当前是关键大拍且确实需要停顿。
· 失败选项的后果不是"扣分+骂一句"就完。要引出新的有趣困境：
  你选了忍→对方觉得你好欺负→下次更过分。你选了打→短期赢了但惹了更大的麻烦。
  每条路都通向新的困境，没有死胡同。
```

#### 【字段叙事效应】（新增——解决字段只是数字的问题）

```
每个数字字段必须定义软阈值效应。格式：
  字段名（范围）：阈值1→效应 | 阈值2→效应 | 阈值3→效应

示例（生成时替换为实际字段）：
  好感度（-100~100）：≤-30→公开敌视 | ≥30→不再敌视 | ≥60→私下帮忙 | ≥80→公开维护 | 100→无法拒绝
  压力值（0-100）：≥50→游戏中体现外在反应 | ≥70→手抖/失眠/沉默 | ≥90→濒临崩溃边缘

AI必须在NPC行为和叙事中体现这些阈值——好感度70的NPC和好感度20时是不同的两个人。
阈值效应不是结局——到100才触发命运转折。但在此之前，数值变化必须有可见的叙事后果。
```

#### 【编辑参考】（新增——追加到 promptBody 末尾）

```
【编辑参考——玩家修改提示词时的格式边界】

当前模板字段清单：
  statusTop: [字段1label], [字段2label], [字段3label]
  taskLine: [字段1label], [字段2label]
  resources: [字段1label], [字段2label]
  variables: [字段1label], [字段2label]

命运转折条件格式：字段名 运算符 数值（多条件用 且/，/、 连接）
支持的运算符：≥ ≤ = > < >= <=
注意：=100 会自动放宽为 ≥95（防止AI回避极端值）
条件括号必须紧挨【命运转折·XXX】标记（≤50字符）

开局格式：开始游戏。【开局编号：N】（N=1~6）
可见成就格式：描述含「字段名+数字阈值」（如"压力值曾达90+但未被压垮"）
隐藏成就触发类型：choice(pattern+count)/gambit(count)/rounds_under(round)/field_zero(字段label)/field_max_under(字段label+阈值)/response_match(pattern+count)
```

### 2.3 修改现有章节

| 章节 | 旧要求 | 新要求 |
|------|--------|--------|
| 【你的身份】 | "AI行为准则。因果结算+选项生成。不替玩家选择" | **加例外条款**："当本回合用户消息末尾出现【★ 命运转折 ★】时，本回合的现状就是该命运转折的场景叙事。命运转折后故事继续，必须给4个选项。" |
| 【世界观】 | 200-350字 | **400-600字**。≥3个命名地点（各1-2句）、≥5个命名NPC（各含动机一句话）、≥2条世界规则（"这个世界里X是不可能的"）。**铁律**："叙事中只能使用 promptBody 已列出的 NPC、地点和规则。禁止即兴创造新势力/新人名/新地点。" |
| 【主角设定】 | 150-250字 | **300-500字**。外貌/能力/身世/处境/可选行动路线 |
| 【核心冲突】 | 自由 | **300-500字**。≥3重困境+核心张力 |
| 【结局系统】 | 4层固定（早/中/后/大后） | **5-8个结局自由设计**。条件括号紧挨标记（≤50字符）。每个结局1-2句具体叙事描述。格式：`结局N·名称（字段≥阈值 且 字段≤阈值）：叙事描述。标注【命运转折·名称】` |
| 【开局系统】 | 3个 | **6个**。每个不同场景+不同初始数值+明确叙事钩子 |
| 【资源系统】 | "2个资源各3获取3消耗" | **加叙事联动**："每个资源变化必须在叙事中有对应事件。获取情报→写出你如何获取的。消耗把柄→写出你用在了谁身上。" |
| 【成就】 | 6-8个含通用3 | **8-12个全部世界观定制**。desc必须含「字段名+数字阈值」。禁止通用名（孤注一掷/长路漫漫/命运之轮由系统兜底自动生成）。按叙事/关系/资源/行为四类分布。 |
| 【隐藏成就】 | 3-5个 | **4-6个**。逐类型给格式示例和参数说明 |

### 2.4 游戏长度联动

| | 短篇 short | 中篇 medium | 长篇 long | 沉浸 immersive |
|------|---------|---------|--------|--------|
| 开局数 | 3-4 | 4-5 | 5-6 | 6 |
| 结局数 | 3-5 | 5-7 | 6-8 | 7-8 |
| 可见成就 | 6-8 | 8-10 | 10-12 | 12-14 |
| 隐藏成就 | 3-4 | 4-5 | 5-6 | 6-7 |
| promptBody预算 | 2000-3000 | 3000-5000 | 4000-6000 | 5000-7000 |
| 结局门槛(回) | 早3-5/中6-10/后10-15/大后15+ | 早5-8/中10-15/后15-25/大后25+ | 早8-12/中15-25/后25-40/大后40+ | 早10-15/中20-30/后35-50/大后50+ |
| 成就"长路"类 | 轮次≥10 | 轮次≥20 | 轮次≥30 | 轮次≥40 |

### 2.5 字段架构弹性化

旧（固定）：statusTop=3, taskLine=2, resources=2, variables=2  
新（弹性）：

| section | min | max | 说明 |
|---------|-----|-----|------|
| statusTop | 2 | 4 | 主角核心状态 |
| taskLine | 2 | 2 | 固定：轮次+主线进度 |
| resources | 1 | 3 | 可消耗资源 |
| variables | 1 | 4 | 关系/态度值 |

要求 AI 按故事需求决定数量——谍战可能需要多个关系值但只需1个资源，生存游戏可能需要3个资源但只需1个关系。

## 三、Phase 2：关键词全局重命名

### 3.1 动机

AI 对"游戏结束"有训练数据惯性——"结束=终止=不能再写"。改成"命运转折"消除这个阻力。

### 3.2 改动清单

| 旧 | 新 | 影响范围 |
|----|-----|---------|
| `【游戏结束·XXX】` | `【命运转折·XXX】` | promptBody、detectEnding()、collectEligibleEndings()、buildEndingInjection()、repairEndingSection()、showEndingOverlay() |
| `【资源不足】` | `【代价沉重】` | promptBody、updateOptionButtons() |
| 结局弹窗标题 | "结局：XXX" → "命运转折：XXX" | showEndingOverlay() |
| 元提示词中所有"结局" | "命运转折" | server.js metaPrompt |
| `buildEndingInjection` 函数名 | `buildTurningPointInjection` | utils.js + core.js 调用处 |
| `checkEndingClientSide` | `checkTurningPointClientSide` | 同上 |
| `detectEnding` | `detectTurningPoint` | 同上 |

### 3.3 兼容性

旧模板、旧存档仍包含旧标记 `【游戏结束·XXX】`。**所有检测函数必须同时兼容新旧两种格式**。新生成的模板只用新格式。

### 3.4 关于字段名

**不改**。"压力值""暴露风险""好感度"在叙事语境里是自然的，改"玉玉"反而出戏。

### 3.5 关于"代价沉重"

这个标记的语义变了——不是"此选项不可用"而是"选了会付出更大代价"。对应的 UI 行为也要改：

**旧行为**：选项按钮变灰，不可点击  
**新行为**：选项按钮可点击，但文字变红/加警告图标，点击时弹出确认："此选择将付出沉重代价，确定？"

这就解决了用户说的"灰掉选项浪费了"的问题——选项还在，玩家可以选，但知道会出事。

## 四、Phase 3：叙事引擎升级

这部分大部分通过 Phase 1 的元提示词新章节落地。但有些需要改代码。

### 4.1 软阈值效应（代码侧）

当前 `checkAchievementsFromState()` 只能检测"≥某值"的成就。软阈值效应（如好感度≥60时NPC行为改变）不需要客户端检测——这是AI在叙事中体现的。但需要确保 promptBody 中定义了这些阈值。

**修法**：元提示词要求后，模板生成时自然包含。无需代码改动。

### 4.2 字段添加时强制叙事定义

在 `addField()` 弹窗中新增强制填写项（见 Phase 5 详细设计）。

### 4.3 initialState 激活

当前模板 JSON 的 `initialState` 是死数据。`startNewGame()` 应读取它初始化 fieldHistory。

**修法**：`startNewGame()` 中，发送开局消息之前：

```javascript
const initialState = tpl.initialState || {};
for (const [fieldId, value] of Object.entries(initialState)) {
    gameState.fieldHistory[fieldId] = {
        current: Number(value) || value,
        max: Number(value) || 0,
    };
}
```

这样开局数值不再由 AI 随机决定，而是模板设计者预设的。

### 4.4 手动改数值后通知AI

玩家在状态栏点数值手动修改后，当前代码只改 `fieldHistory` 不通知 AI。修法：改完后追加一条 user 消息到 fullHistory：

```javascript
gameState.fullHistory.push({
    role: 'user',
    content: '【系统通知】玩家手动调整了「' + fieldLabel + '」的值为 ' + newValue + '。请在后续叙事中合理衔接。'
});
```

## 五、Phase 4：结局指令强化

### 5.1 改为独立 system 消息

`_prepareMessages()` 中，`buildTurningPointInjection()` 的返回不再拼到用户消息里，而是追加为独立 system 消息：

```javascript
// 旧（core.js :285）
enhancedContent = enhancedContent + '\n\n' + endingInjection;

// 新
if (preEndingType) {
    allMessages.push({ role: 'system', content: endingInjection });
}
```

放在消息数组最后一条 system 位置，user 消息之前——覆盖效应让 AI 无法忽视。

### 5.2 文案优化

```
【★ 命运转折回合 ★】
本回合的现状就是「XXX」的场景。不需要额外切换。
结算上回合后果后，现状直接写命运转折叙事（8-12句）。
末尾输出【命运转折·XXX】。然后照常给4个选项。
```

关键变化：
- 不再说"必须触发"，而是"现状就是"——AI 不需要额外做什么
- "不需要额外切换"——解除 AI 的任何规则顾虑
- 用新标记名【命运转折】

### 5.3 元提示词模板生成时的例外条款

`【你的身份】` 章节强制包含：

> "当本回合出现【★ 命运转折回合 ★】系统消息时，本回合的现状就是该命运转折的场景叙事。命运转折后故事继续，必须给4个选项。"

这样系统提示词本身就有例外条款，AI 不会觉得命运转折和叙事规则冲突。

## 六、Phase 5：编辑系统校验

### 6.1 字段重命名/删除 → promptBody 引用断裂

**场景**：玩家在字段编辑器把 "压力值" 改成 "精神压力" → promptBody 里的 `（压力值=100）` 全变死引用。

**修法 A**：`saveFields()` 新增扫描逻辑

```
1. 收集旧label → 新label 映射（利用 _preEditFields 快照）
2. 在 promptBody 中全文搜索每个旧label
3. 找到匹配 → 弹窗："检测到 promptBody 中有 N 处引用了「旧label」。是否全部替换为「新label」？"
4. 确认 → 自动替换。拒绝 → 保留（玩家可能故意在叙事中保留旧名）
```

**修法 B**：`savePrompt()` 新增字段引用校验

```
1. 扫描 promptBody 中所有（XXX运算符数字）格式的条件
2. 提取 XXX（字段标签）
3. 逐一匹配 outputSections 中所有字段的 label
4. 不匹配的 → 弹窗警告列表："以下条件引用了不存在的字段：XXX。这些命运转折将无法触发。"
```

### 6.2 添加字段时强制叙事定义

当前 `addField()` → 给空壳 `{label:'新字段', icon:'📌', type:'number'}` → AI 不知道这是什么。

**修法**：添加字段弹窗包含：

```
字段ID：________
显示标签：________
图标：🎨 [选择]
类型：[number/text]
格式提示：________

📝 叙事含义（必填——AI 据此理解字段在故事中的含义）：
________________________________
________________________________

软阈值（可选）：
  阈值1：___ → 效应：___________
  阈值2：___ → 效应：___________
```

保存时：
1. 字段写入 outputSections
2. 叙事含义自动追加到 promptBody 为 `【字段定义·XXX】` 章节
3. 修复 Phase 1 的【编辑参考】字段清单

### 6.3 隐藏成就 trigger 编辑器

当前 `addNewAchievement(true)` 硬编码 `trigger: { type: 'gambit', count: 1 }`。

**修法**：添加隐藏成就弹窗增加：

```
触发类型：[下拉：choice / gambit / rounds_under / field_zero / field_max_under / response_match]
```

根据类型显示对应参数输入：

| 类型 | 参数1 | 参数2 |
|------|-------|-------|
| choice | 匹配词(pattern) | 次数(count) |
| gambit | 次数(count) | — |
| rounds_under | 回合数(round) | — |
| field_zero | 字段(下拉选择已有字段) | — |
| field_max_under | 字段(下拉) | 阈值(threshold) |
| response_match | 匹配词(pattern) | 次数(count) |

### 6.4 「恢复原始」范围选择

当前 `resetPrompt()` 只恢复 promptBody，不恢复字段/成就。

**修法**：弹窗多选

> "恢复原始设定：[×] 提示词正文 [×] 字段结构 [×] 可见成就 [×] 隐藏成就"

默认全选。确认后一次性恢复所有选定项。

### 6.5 游戏中编辑模板 → 警告

`saveFields()` 和 `savePrompt()` 中检测 `gameState.gameStarted`：

> "⚠ 当前游戏已进行 N 回合。修改模板设定后，已有的对话历史可能与新设定不一致。建议开新游戏。确定继续修改？"

### 6.6 服务端模板同步

当前 `saveFields()` 只写 localStorage。内置模板的服务端磁盘文件是旧版。

**修法**：`saveFields()` 成功后 POST `/api/templates/:id` 同步服务端。

## 七、Phase 6：全局状态隔离

### 7.1 AI 指令按存档隔离

| 当前 | 改为 |
|------|------|
| `LS_KEYS.aiInstructions = 'xixi_ai_instructions'` | `LS_KEYS.aiInstructions(id) → 'xixi_ai_instructions_' + id` |

- `getAiInstructions()` 读取当前存档的指令
- `saveAiInstructions()` 写入当前存档
- `clearAiInstructions()` 清除当前存档
- `continueGame()` 不额外清理——每个存档独立
- `startNewGame()` 清除当前存档的指令

### 7.2 自定义图片按存档隔离

| 当前 | 改为 |
|------|------|
| `LS_KEYS.customImages = 'xixi_custom_images'` | `LS_KEYS.customImages(id) → 'xixi_custom_images_' + id` |

### 7.3 AI 指令合并到提示词 → 永久污染

`mergeInstructionsToPrompt()` 合并前加强警告：

> "将 N 条指令永久写入模板。所有使用此模板的存档（包括新游戏）都将包含这些规则。此操作不可撤销。确定？"

合并完后清空当前存档的 AI 指令列表。

## 八、Phase 7：数据安全

### 8.1 「新游戏」静默删除全部存档

`selectSave()` (`ui.js:577`) → `for s=0..9 removeItem(getSaveKey(saveId,s))` 无确认。

**修法**：如果该模板有存档进度，弹窗：

> "「XXX」已有游戏进度（N个存档槽位，最新第M回合）。开始新游戏将清除所有进度。确定？"

确认后才删除。

### 8.2 「恢复原始提示词」无确认

`resetPrompt()` 无确认直接覆盖。

**修法**：弹窗确认（见 6.4——合并到范围选择弹窗）。

### 8.3 命运转折后继续 → AI 不知道已触发

`closeEndingOverlay()` 只关弹窗。下一回合 AI 不知道命运转折已完成。

**修法**：关闭弹窗时追加一条 system 消息到 fullHistory：

```javascript
gameState.fullHistory.push({
    role: 'user',
    content: '【系统通知】命运转折「' + endingType + '」已触发。故事继续，请推进后续剧情。不要再次触发同一命运转折。'
});
```

配合 `triggeredEndings` 数组的客户端兜底，双重保障。

## 十、Phase 9：服务端统一

### 10.1 两个 buildSystemPrompt 不一致

`server.js:107-126` 和 `utils.js:161-191` 各有一套。服务端版本缺安全边界章节。

**修法**：抽成共享模块。或者在服务端版本补上安全边界。确保两个版本输出完全一致。

### 10.2 两个 repairEndingSection 实现重复

`server.js:132-181` 和 `utils.js:113-154` 各一份。

**修法**：只保留一份权威实现。客户端调用服务端版本（通过 API），或者把逻辑统一到一个地方。

### 10.3 摘要质量

当前 `JSON.stringify(messages)` 把对话对象序列化后发给AI——AI 看到的不是自然对话而是 JSON 文本。

**修法**：格式化为自然对话格式再发送：
```
玩家选择了：忍气吞声 — 代价...
AI回应：你低下头...
玩家选择了：反击 — 代价...
```

### 10.4 token 用量追踪

当前只按回合数触发摘要（>20条未摘要消息）。但如果 AI 回复很长，token 可能远超上下文窗口。

**修法**：粗略估算 token 数（中文 ~1.5 字符/token）。在 `maybeSummarize()` 中加入 token 阈值检查。

## 十一、Phase 10：UI 修复

### 11.1 手机版 sunset 主题按钮掉落

问题：`themes/theme-sunset.css` 中某处 CSS 在移动端导致存档按钮换行溢出。

**修法**：检查 sunset 主题的 `.btn` / `.btn-small` / `.btn-tiny` 是否锁死了尺寸，以及 `.top-actions` 的 flex-wrap 行为。参考其他已修复的主题（森林、修仙、赛博都补过响应式断点）。

### 11.2 UI 命名：「存档」→「模板」

| 旧名 | 新名 | 出现位置 |
|------|------|---------|
| "我的存档" | "我的模板" | 标签页标题 |
| "创建新存档" | "创建新模板" | 按钮文字 |
| "存档列表" | "模板列表" | 面板内标题 |
| "存档名称" | "模板名称" | 创建表单 |
| 槽位1-9 | 保持"存档槽位" | 这些是进度存档，不是模板 |

区分：大的（世界设定+提示词）= **模板**，小的（游戏进度）= **存档**。

### 11.3 字段编辑器图标双显

当前每个字段行有两个图标元素：`<input class="field-icon">` 和 `<button class="btn-pick-emoji">`。两者显示同一个 emoji，视觉上是两个。

**修法**：
1. `saveFields()` 改用 class 选择器（`.field-id`、`.field-label`、`.field-icon`、`.field-format`）代替位置索引 `inputs[0..3]`
2. icon input 改为 `type="hidden"`
3. 保留按钮作为唯一的可见图标元素
4. emoji 选择器选择后同时更新隐藏 input 的值 + 按钮显示

## 十二、Phase 11：新功能

### 12.1 难度选择

新建模板时增加难度选项：简单 / 标准 / 困难 / 噩梦。

影响元提示词：
- **简单**：NPC 敌意低，资源充裕，选项代价小，软阈值偏向有利
- **困难**：NPC 敌意高，资源稀缺，选项代价大，命运转折触发更易
- **噩梦**：每题都是陷阱，没有安全选项

在 promptBody 的【资源系统】和【选项设计】章节中体现难度差异。

### 12.2 从故事/提示词创建模板

替代填表流程：粘贴一段故事概要或已有提示词 → AI 反向解析 → 生成模板 JSON → 进入预览编辑。

新增 API：`/api/parse-story`，接受文本输入，调用 DeepSeek 解析为模板结构。

酒馆使用场景：玩家从酒馆下载了旧模板，想更新但不想重新填表 → 导出旧模板的 promptBody → 粘贴到解析器 → AI 重新生成规范化模板。
